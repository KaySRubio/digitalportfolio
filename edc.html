Examples of EDC Problems I've Solved

Repeating Forms with Repeating Rows


Challenge

During EDC design for a longitudinal study, a sponsor requested that we create a medication CRF based on a paper form that listed 10 medications per form, and would be completed at each time point. The form would be repeating since some participants could have dozens of medications.

Solution

Based on prior experience, I knew this would create a tricky exported data structure and unnecessary duplicate data for subjects who stay on the same medication across time points. I explained to the sponsor that it would be better to have a repeating medication form with one medication only inside a special "Medications" time point, rather including the medication form at each time point in the database.

This ensured that if participants stayed on the same medications throughout the study, the site wouldn't have to re-enter them at every time point, but could instead enter each medication once and then update start/stop dates over time. It also would avoid a weird data structure that happens when there are repeating rows on a repeating forms.

For instance, if the form was designed to be repeating and include 10 medications per form as requested, participant A with 13 medications would have two different rows in the exported data. Medications 1-10 would be in the columns of the first row, and medications 11-13 would be in the columns of the second row.

This is hard for statisticians and data managers to analyze. It's possible to use SQL re-format the data to a long format file, where participant A has 13 rows, one medication per row. However, it's easier to set up the eCRF's in a way that automatically exports in that format, rather than having to transform the data later.

I provided some screenshots showing the proposed form setup to the sponsor, and they agreed that this structure would be easier for sites and easier for future data analysis.

Drug Dispensing Challenge


Challenge

A ongoing medication trial had a change of supplier, and the bottles changed from 30 pills (lasting 15 days) to 60 pills (lasting 30 days). The participants needed to re-consent to a protocol amendment before being allowed to take the new bottles. When sites request a dispense, the EDC assigns the bottle ID(s) to the participant. The EDC needs to be "smart" enough to continue to assign the old bottles to participants who have not re-consented, and assign the new bottles to those who have.

Solution

I made a copy of the live database and uploaded the list of new bottle ID's as a new batch with the number of treatment days per bottle as 30 instead of 15. The EDC uses several logical factors to assigns the bottle ID(s) to certain participants.

The existing factors pertained to randomization assignment, consent at baseline, and number of bottles requested. In order to accommodate the new batch, I programmed new rules; if Re-Consent = False, the old batch will be dispensed. If Re-Consent =True, the new batch would be dispensed.

To make sure it worked, checked it myself, and set up a UAT script so a colleague could perform formally documented UAT. I also created slides to train sites about the new relationship between the Re-Consent page and the dispense features. Once UAT was complete and the PM approved the changes, I merged the changes into the live database.

Mid-Study Update Challenge


Challenge

A co-worker was running into errors while trying to update her study's EDC system and asked me for help. Her project manager had requested that she delete certain variables that were no longer needed, and change the question text of other variables or codelists.

Solution

I explained that deleting variables can orphan existing data, and changing the question text away from the original question's essential medical meaning can render the existing data incorrect.

I re-trained her to hide the variables rather than deleting them, and create new variables instead of significantly changing the existing question or codelist text.

I also checked the exported data to see what had already been entered, and provided the PM with information how many data points would need to be re-done by the sites after the update.
